\subsection{Usability}
This subsection outlines the usability testing plan for non-functional
requirement NFR6.2.

The current implementation includes packet-level support for source priority,
but project-level priority definitions and decision rules are not finalized.
Because of this, formal validation of NFR6.1 is deferred and not included in
this usability test cycle.

\textbf{Evaluation scope:}
\begin{itemize}
  \item \textbf{Non-intrusive display (NFR6.2):}
  confirm that users can continue normal visual tasks without major
  obstruction or discomfort.
\end{itemize}

\textbf{Usability testing procedure:}
\begin{enumerate}
  \item Recruit at least 4 stakeholders from the target user group.
  \item Provide a short orientation on HUD indicator meaning and test steps.
  \item Run directional cue scenarios during typical visual tasks (walking,
  reading signs, and obstacle awareness simulation) to assess NFR6.2.
  \item Record observations on cue correctness, user hesitation, distraction,
  and discomfort reports.
  \item Collect post-session survey ratings (1--5 scale) for
  non-obtrusiveness, clarity, and situational awareness.
\end{enumerate}

\textbf{Planned acceptance criteria:}
\begin{itemize}
  \item NFR6.2: average non-obtrusiveness rating is at least 4.0/5.0.
  \item NFR6.2: no stakeholder reports significant visual obstruction or
  discomfort during normal task flow.
  \item Minimum participant count requirement is met ($n \geq 4$).
\end{itemize}

\section{Changes Due to Testing}

\subsection{General Feedback}

Usability testing was completed with four stakeholders using the plan in
Section 4.2. The average user rating was \textbf{4.75/5.0} across
non-obtrusiveness, clarity, and situational awareness, and all planned metrics
were met.

The main feedback item was that the visualization scale looked a bit too large
in some tasks. Based on this, we scaled the visualization down to reduce visual
load while keeping cues easy to read.

In addition, testing discussions highlighted the need for better behavior during
unexpected failures. We are implementing system fault handling so that if the
system crashes, it can fail safely and recover in a controlled way.

Moreover, the classification module was identified as an area for improvement 
in terms of reliability and accuracy, which led to the improvements mentioned 
in the next section. 

\subsection{Classification Module}
After user testing, it was found that the classification module could be 
improved to provide more reliable output the the user. Specifically related to 
the confidence level and overall accuracy of the classification results. To 
address these issues, the classification module was updated as follows:

\subsubsection{Confidence Implementation}
The classification module was updated to calculate the confidence level of each 
class prediction. This was achieved by implementing a softmax function to 
convert the raw output of the classifier into probabilities. The confidence 
level is determined by the highest probability among the predicted classes, 
providing feedback to the overall system of how much trust to put in the 
classification results. These metrics were used to improve the overall 
reliability of the system by filtering out low confidence predictions and 
substituting them with an "Unknown" classification when the confidence is 
below a current threshold of 70\%. This threshold was determined by manually 
analyzing the confidence levels of correct and incorrect classifications during 
testing with the current PCA and LDA trained matricies. However, to meet the 
requirements specified in the VnV plan, the classification module had to 
further be improved so that the confidence threshold can be increased to 90\% 
or higher. This led to the next improvement mentioned below. 

\subsubsection{Classification Accuracy Improvement}
The accuracy of the classification module was low because of two main reasons: 
limited examples in the training dataset and choosing classes that had very 
similar audio signatures (car horn vs siren). To address these issues, the 
first change that was made was finding another sound dataset that had more 
examples and more distinct classes to choose from. The dataset that was chosen 
was the FSD50k sound dataset, which has over 50,000 audio samples across 200
classes \cite{FSD50k}. Using this dataset, the second change that was made was 
changing the list of classes that the system was trained to classify to have 
more distinct audio signatures. The new classes that are chosen are Chatter, 
Bicycle Bell, Siren and Knock.

\section{Automated Testing}

To ensure consistent verification throughout development, multiple automated
testing CI pipelines were implemented using GitHub Actions. These pipeline are
triggered on each push to the main branch, and any Pull requests targetted to
the main branch. Pull requests are blocked from merging if any pipeline fails
to ensure verification holds for any new changes to the project.
These automation testing reduces reliance on manual testing and increases the
repeatability and reliability of the verification process.

The automated workflow are as follows:

\begin{itemize}
  \item \textbf{Build Verification:} The source code is compiled using hardware
    compiler to ensure that all changes integrate correctly and do not
    introduce build errors.

  \item \textbf{Unit Testing:} All unit tests are executed using gtest
    \cite{GoogleTest} to verify the functional correctness of core software
    components.
  \item \textbf{Static Code Analysis:} A C++ static analysis tool
    \cite{clangStaticAnalyzer} is run to detect potential programming errors,
    memory issues, and violations of coding standards.
\end{itemize}


The results of each automated pipeline are recorded and made available through
the \href{https://github.com/Team6-SixSense/audio360/actions}{repository's GitHub Actions tab}
for traceability and review.

libraries are well-tested by the C++ community.
