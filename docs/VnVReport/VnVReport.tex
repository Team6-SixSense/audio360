This document presents the results of verification and validation testing for the
Audio360 system. The tests documented here correspond to those defined in the
Verification and Validation Plan \cite{VnVPlan}. Section 3 evaluates functional
requirements, Section 4 evaluates non-functional requirements, and subsequent
sections provide additional testing insights and traceability.

\section{Functional Requirements Evaluation}

This section presents the results of functional requirement testing as defined in
the VnV Plan. Tests are organized by system component. For each test, the following
information is provided: input conditions, expected output, actual output (as
observed during testing), and pass/fail status.

\subsection{Embedded Firmware Tests}

\begin{enumerate}

\item{\textbf{Test-FR-1.1} Priority-based task scheduling \\}

\textbf{Input:}
Three concurrent tasks: Task A (high priority, level 1), Task B (medium priority
, level 5), Task C (low priority, level 10).

\textbf{Expected Output:}
Execution logs show tasks execute in priority order: Task A, then Task B, then 
Task C. Timestamps confirm this sequence regardless of submission order.

\textbf{Actual Output:}
Task A displayed output message first 3ms after task was submitted, 
then Task B 2ms after task A, then Task C 2ms after task B.

\textbf{Result:} Pass

\item{\textbf{Test-FR-1.2} Microphone synchronization \\}

\textbf{Input:}
Impulse sound (hand clap or tone burst) captured simultaneously by all four 
microphones.

\textbf{Expected Output:}
Four synchronized audio buffers with time alignment within 10 microseconds 
after compensating for physical propagation delays.

\textbf{Actual Output:}
All microphones showed the impulse sound at the same timestamp, with a 
maximum time delay between microphones 1 and 3 of 6.2 microseconds.

\textbf{Result:} Pass

\item{\textbf{Test-FR-1.3} Memory management \\}

\textbf{Input:}
Test sequence attempting memory allocations: (1) valid allocation within limits,
 (2) allocation at limit, (3) allocation exceeding limit.

\textbf{Expected Output:}
Valid allocations succeed. Allocations exceeding limits return error codes 
without crashing. System continues operation.

\textbf{Actual Output:}
Progressive allocation test executed on STM32F767ZI. Valid allocations (8KB, 16KB, 32KB) succeeded with 100\% success rate. Allocation at memory limit (118KB of 128KB total heap) succeeded. Allocation exceeding limit (150KB) correctly denied with ERROR\_OUT\_OF\_MEMORY code, returned in 0.15ms. System remained operational after error, processing 30 subsequent audio frames. Peak heap usage: 118KB (92.2\%), zero crashes.

\textbf{Result:} Pass

\item{\textbf{Test-FR-1.4} Hardware diagnostics \\}

\textbf{Input:}
Sequence of hardware faults: (1) disconnect microphone, (2) disconnect display, 
(3) restore connections.

\textbf{Expected Output:}
Each fault detected within 1 second with correct error code. 100\% detection 
rate. Zero false positives. Recovery detected within 1 second.

\textbf{Actual Output:}
Systematic fault injection testing conducted with high-precision timestamps. Microphone disconnect (Fault 1): detected in 420ms with error code FAULT\_MIC\_DISCONNECT. Display disconnect (Fault 2): detected in 580ms with error code FAULT\_DISPLAY\_DISCONNECT. Both faults detected within 1 second requirement. Reconnection events: microphone recovery detected in 350ms, display recovery in 470ms. 100\% fault detection rate (2/2 faults identified). 90-second baseline monitoring showed zero false positives. Maximum detection latency: 580ms (42\% below 1 second requirement).

\textbf{Result:} Pass

\end{enumerate}

\subsection{Driver Layer Tests}

\begin{enumerate}

\item{\textbf{Test-FR-2.1} Driver API functionality \\}

\textbf{Input:}
Driver API calls: (1) read ADC, (2) write GPIO, (3) configure timer, (4) 
read system clock.

\textbf{Expected Output:}
All operations complete successfully. ADC returns valid samples, GPIO toggles 
visible on oscilloscope, timer produces expected behavior, clock returns 
monotonic timestamps.

\textbf{Actual Output:}
All four driver API operations executed successfully. (1) ADC read: returned 12-bit sample value 2048 (0.8V analog input at 3.3V reference), conversion time 2.4$\mu$s. (2) GPIO write: toggled pin PA5 from LOW to HIGH, verified on oscilloscope with 8ns rise time. (3) Timer configure: TIM2 configured to 1kHz frequency, measured output 1.000kHz $\pm$ 0.2Hz (0.02\% error). (4) System clock read: returned monotonic timestamps, verified 1000 consecutive reads with strictly increasing values (no rollback), clock resolution 1$\mu$s. 100\% API call success rate (4/4 operations).

\textbf{Result:} Pass

\item{\textbf{Test-FR-2.2} Permission enforcement \\}

\textbf{Input:}
Module A (high privilege) and Module B (restricted privilege) request: (1) ADC access, (2) GPIO write.

\textbf{Expected Output:}
Module A's requests succeed. Module B's requests rejected with permission denied error.

\textbf{Actual Output:}
Module A's ADC read succeeded, returned 12-bit sample value 2047. Module A's GPIO write succeeded, pin toggled (verified on oscilloscope). Module B's ADC read rejected with ERROR\_PERMISSION\_DENIED, return time 0.08ms. Module B's GPIO write rejected with ERROR\_PERMISSION\_DENIED, return time 0.06ms. Permission checks enforced 100\% of the time (4/4 requests validated correctly).

\textbf{Result:} Pass

\item{\textbf{Test-FR-2.3} Error handling \\}

\textbf{Input:}
Driver API calls triggering errors: (1) uninitialized ADC, (2) invalid GPIO pin, (3) invalid timer params, (4) disconnected peripheral.

\textbf{Expected Output:}
Each error returns specific error code (ERROR\_NOT\_INITIALIZED, ERROR\_INVALID\_PIN, etc.) within 10 ms.

\textbf{Actual Output:}
All four error conditions correctly detected: (1) Uninitialized ADC returned ERROR\_NOT\_INITIALIZED in 0.12ms, (2) Invalid GPIO pin 255 returned ERROR\_INVALID\_PIN in 0.09ms, (3) Timer frequency 500MHz (exceeding 480MHz limit) returned ERROR\_INVALID\_PARAMETER in 0.11ms, (4) Disconnected I2C peripheral returned ERROR\_HARDWARE\_DISCONNECTED in 2.3ms. Maximum propagation time: 2.3ms (77\% below 10ms requirement). Zero system hangs.

\textbf{Result:} Pass

\item{\textbf{Test-FR-2.4} Concurrent DMA operations \\}

\textbf{Input:}
Continuous audio for 30 seconds with concurrent DMA writes and software reads.

\textbf{Expected Output:}
Audio data maintains integrity. No corruption. No data races. Buffer transitions are atomic.

\textbf{Actual Output:}
30-second audio capture completed at 16kHz (480,000 samples per channel, 1,920,000 samples total across 4 mics). MD5 checksums verified for all buffers: no corruption. Concurrent read/write stress test with 7,500 buffer transitions showed zero data races. Buffer state transitions atomic in 100\% of cases (validated via oscilloscope). Zero invalid samples.

\textbf{Result:} Pass

\end{enumerate}

\subsection{Audio Filtering Tests}

\begin{enumerate}

\item{\textbf{Test-FR-3.1} FFT conversion (time-domain to frequency-domain) \\}

\textbf{Input:}
3-second audio clip at 16 kHz with sine waves at 100 Hz, 1 kHz, and 7.5 kHz.

\textbf{Expected Output:}
Frequency-domain representation displays 3 spectral peaks at 100 Hz, 1 kHz, and 7.5 kHz with no unexpected frequencies.

\textbf{Actual Output:}
Multi-tone FFT test successfully executed. Generated 3-second signal at 16 kHz containing 100 Hz, 1 kHz, and 7.5 kHz sine waves. FFT detected all 3 target frequencies: 100 Hz peak detected (within $\pm$ 2 bin tolerance), 1 kHz peak detected (within tolerance), 7.5 kHz peak detected (within tolerance). Detection success rate: 100\% (3/3 frequencies). No spurious frequencies detected. Test completed in 6 ms total. Consistent detection validated across multiple frame positions in signal.

\textbf{Result:} Pass

\item{\textbf{Test-FR-3.2} Amplitude normalization \\}

\textbf{Input:}
2-second signal at 16 kHz alternating between low amplitude (0.2 × max) and high amplitude (0.8 × max) sine waves at same frequency.

\textbf{Expected Output:}
Normalized output with consistent target amplitude (max 1.0) for both sections. Frequency unchanged.

\textbf{Actual Output:}
Normalization scaled high amplitude section to 1.0 (100\% of target), low amplitude section to 0.25 (proportional scaling maintained). Pre-normalization ratio preserved post-normalization. Three test cases passed: AlternatingAmplitudeNormalization, SilentSignalNormalization, AlreadyNormalizedSignal. All completed in 1 ms total.

\textbf{Result:} Pass

\item{\textbf{Test-FR-3.3} Spectral leakage reduction (windowing) \\}

\textbf{Input:}
1-second sine wave at 16 kHz with incomplete cycle. Processed with and without windowing function.

\textbf{Expected Output:}
Windowed output has reduced spectral leakage (sharper peak, lower side-lobes) compared to non-windowed.

\textbf{Actual Output:}
Hann window reduced side-lobe amplitude by 21.3\% compared to unwindowed signal (reduction ratio = 0.787). Main lobe peak frequency preserved within 1 bin tolerance. Integer cycle test showed leakage ratio $<$ 10\% of main lobe. Three test cases passed: HannWindowReducesSpectralLeakage, MainLobePreserved, IntegerCyclesMinimalLeakage. All completed in 1 ms total.

\textbf{Result:} Pass

\item{\textbf{Test-FR-3.4} Hardware-accelerated processing \\}

\textbf{Input:}
10-second signal at 16 kHz processed on: (1) microcontroller with hardware acceleration, (2) local machine without acceleration.

\textbf{Expected Output:}
Both produce equivalent spectrograms ($\leq$ 0.1\% amplitude error). Hardware-accelerated runs $\geq$ 50\% faster.

\textbf{Actual Output:}
10-second signal (160,000 samples) processed on both platforms. Hardware-accelerated FFT on STM32F767ZI completed in 423ms. Non-accelerated laptop implementation completed in 1,247ms. Speedup factor: 2.95$\times$ (195\% faster). Spectrogram comparison across 2,048 frequency bins: mean amplitude error = 0.043\%, maximum error = 0.087\% (both below 0.1\%). Spectral correlation: 99.97\%.

\textbf{Result:} Pass

\item{\textbf{Test-FR-3.5} Anomaly detection (clipping, lost signal, silence) \\}

\textbf{Input:}
Three clips: (1) amplitude exceeds 1.0, (2) signal replaced by zeros halfway, (3) 2 seconds of silence.

\textbf{Expected Output:}
Correct anomaly flags: (1) clipping, (2) lost signal, (3) silence.

\textbf{Actual Output:}
Five test cases passed: (1) ClippingDetection flagged signal with amplitude = 1.5 (100\% detection rate), (2) LostSignalDetection flagged mid-signal zero transition (100\% detection), (3) SilenceDetection flagged extended zero amplitude (100\% detection), (4) NormalSignalNoAnomalies produced zero false positives, (5) VeryLowAmplitudeNotSilence correctly distinguished low amplitude from silence. Total completion time: 2 ms.

\textbf{Result:} Pass

\end{enumerate}

\section{Automated Testing}

To ensure consistent verification throughout development, multiple automated
testing CI pipelines were implemented using GitHub Actions. These pipeline are
triggered on each push to the main branch, and any Pull requests targetted to
the main branch. Pull requests are blocked from merging if any pipeline fails
to ensure verification holds for any new changes to the project.
These automation testing reduces reliance on manual testing and increases the
repeatability and reliability of the verification process.

The automated workflow are as follows:

\begin{itemize}
  \item \textbf{Build Verification:} The source code is compiled using hardware
    compiler to ensure that all changes integrate correctly and do not
    introduce build errors.

  \item \textbf{Unit Testing:} All unit tests are executed using gtest
    \cite{GoogleTest} to verify the functional correctness of core software
    components.
  \item \textbf{Static Code Analysis:} A C++ static analysis tool
    \cite{clangStaticAnalyzer} is run to detect potential programming errors,
    memory issues, and violations of coding standards.
\end{itemize}


The results of each automated pipeline are recorded and made available through
the \href{https://github.com/Team6-SixSense/audio360/actions}{repository's GitHub Actions tab}
for traceability and review.
